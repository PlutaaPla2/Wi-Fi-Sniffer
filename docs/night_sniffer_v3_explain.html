Overview

night_sniffer_v3.py is a production-ready occupancy estimation script. By monitoring 802.11 management frames (probes, associations, auths), it isolates clients from access points, calculates signal-to-distance buckets, and handles device count correction through active MAC session management.

Configuration Parameters

The program is controlled by a set of hardcoded parameters at the top of the file:

Setting Value Explanation
INTERFACE "wlan1" Target Wi-Fi interface in monitor mode.
LOG_FILE "wifi_full_recon_report.csv" Main raw capture CSV target.

CHANNEL_HOP_INTERVAL 0.5 Seconds to stay on each channel when channel hopping.

Distance and Room Proximity Buckets

Using a log-distance path-loss model, the RSSI (Received Signal Strength Indicator) is converted to meters:

Distance = 10 ^ ((P0 - RSSI) / (10 * N))

Where P0 = -35 is the reference RSSI at 1 meter, and N = 3.0 is the indoor attenuation path-loss exponent. The calculated distance is classified into coarse zones:

  • immediate: ≤ 2 meters
  • near: ≤ 7 meters
  • mid: ≤ 20 meters
  • far: > 20 meters

MAC Identification & Verification

MAC addresses provide a primary identifier for Wi-Fi frames, but modern mobile devices rotate MAC addresses to preserve privacy. The sniffer applies multi-stage inspection to determine MAC status and extract device characteristics.

1. MAC Type Categorization

In check_mac_type(), the script isolates the first byte (octet) of the source MAC address and performs a bitwise check on the second least significant bit (known as the Universal/Local bit or U/L bit): 

# Extract first byte, e.g., 'da:a1:19...' -> da (base 16) -> 218 (base 10)
first_byte = int(mac.split(":")[0], 16)
is_randomized = bool(first_byte & 0x02)

Bit layouts for the first octet of MAC addresses are defined as follows:

Bit 7..2 Bit 1 (U/L Bit) Bit 0 (I/G Bit)
OUI / Manufacturer Bits 0 = Universally Administered (Real MAC)
1 = Locally Administered (Randomized MAC) 		0 = Unicast Address
1 = Multicast / Broadcast Address

If the U/L bit is 1, the MAC address is classified as "Randomized", meaning it was dynamically generated by the operating system (iOS, Android, Windows) and does not match a registered physical NIC vendor.

2. OUI Lookup & Database Queries

For MACs labeled as "Real", the script queries the local MacLookup vendor database. If the lookups fail or if the MAC is randomized, the script assigns a placeholder of "Randomized/Unknown" or "Unknown" to avoid polluting vendor metrics.

3. Heuristic Identity Correlation

In get_correlation_identity(), the script attempts to reconstruct a device's identity by digging into the frame's Information Elements (IEs):

  • Vendor Specific Elements (Tag 221): The script loops through Dot11EltVendorSpecific layers. It extracts the OUI integer (3 bytes) and matches it against known hardware manufacturers. For example, if 0x0017F2 (Apple OUI) is found inside vendor-specific tags, the device is flagged as an Apple device even if the outer source MAC address is randomized.
  • Extended Supported Rates (Tag 50): The script inspects the supported frequency rates listed inside Tag 50. If channels 12 or 13 are present in the list, the device's locale is identified as TH/EU (since channels 12 and 13 are not permitted for active probing in the US). Furthermore, if the supported rates list contains more than 11 entries, it indicates a modern, dual-band, or high-throughput interface, prompting a device class categorization of "High-End" (as opposed to "IoT/Low-End").

Information Element (IE) Fingerprinting

Information Elements (IEs) are structured variable-length parameters appended to 802.11 management frames. They advertise capabilities (such as supported speeds, encryption, vendor extensions, HT, VHT, and WPS).

The Scapy Parsing Limitation & Custom Solution

Scapy's default Dot11Elt structure parses elements recursively as chained layers. However, if Scapy encounters a malformed element, an undocumented vendor extension, or a parsing error, it frequently stops decoding subsequent elements. To solve this, night_sniffer_v3.py bypasses Scapy's object chain and manually walks the raw byte array of the elements in extract_ie_details(): 

# Cast the first element to bytes to get the full raw payload chunk
raw_bytes = bytes(first_elt)
i = 0
while i + 1 < len(raw_bytes):
    ie_id = raw_bytes[i]
    ie_len = raw_bytes[i + 1]
    data_start = i + 2
    data_end = data_start + ie_len
    if data_end > len(raw_bytes):
        break
    info = raw_bytes[data_start:data_end]
    
    # Process information element ...
    i = data_end

Fingerprint Extraction Logic

As the parser loops through the raw payload, it extracts four critical telemetry features:

  1. IE Sequence: An ordered list of all IE IDs (e.g., 0,1,50,45,221,221) present in the frame. This represents the sequence in which the device constructs its capability parameters.
  2. Capabilities: Detects specific, standardized protocol features by checking IDs:
    • 45 → HT (High Throughput, 802.11n)
    • 48 → RSN (Robust Security Network, WPA2/WPA3)
    • 50 → EXT_RATES (Extended Rates)
    • 191 or 192 → VHT (Very High Throughput, 802.11ac)
    • 255 → EXT_CAP (Extended Capabilities)
    • 221 & \x00\x50\xf2\x04 → WPS (Wi-Fi Protected Setup)
  3. Vendor IEs: For Vendor Specific Elements (Tag 221), the script extracts the first 3 bytes of the data payload, formatting it as a colon-separated OUI string (e.g. 00:50:f2). These are grouped and sorted to isolate device-specific chipsets.
  4. IE Fingerprint: For each parsed IE, a string is constructed in the format: "{ie_id}:{len(info)}:{first_8_bytes_hex}". All strings are joined with a pipe character (|) and hashed via SHA-1: 
    # Example input string for SHA-1 hashing:
# "0:9:4d7957694669|1:8:82848b960c12|50:4:b0b1b2b3"
ie_fingerprint = hashlib.sha1(raw_fingerprint.encode("ascii")).hexdigest()[:16]
    The resulting 16-character hexadecimal signature is extremely stable across rotated MAC addresses. Because a device's OS and network chip features remain identical during MAC rotation, their capability payloads match, permitting tracking correlation.

Randomized MAC Session Scoring

To avoid overcounting users who rotate their MAC address, a scoring system matches incoming randomized frames to active tracking sessions:

Factor Score Delta Description
Same IE Fingerprint +3 Perfect capability match
Different IE Fingerprint -3 Capability mismatch
Vendor IE Overlap +2 Shared vendor-specific tags
Probe SSID Overlap +2 Device probing for same network names
Close RSSI (≤ 6dBm) +2 Signals are at similar strengths
Same Proximity Zone +1 Matches spatial context
Nearby Time Window (≤ 120s) +1 Close temporal context
Threshold logic: A session merges if the total score is ≥ 6, and at least 2 independent supporting reasons are satisfied.

Packet Handler

handle_packet() inspects incoming frames and acts on client management subtypes (Probe Requests, Associations, Auth/Deauth, Disassociation). For client frames, it updates active session tracking, prints a color-coded log line in the terminal, and writes full properties to the CSV database.

Background Workers

The script handles two background tasks concurrently:

  • Channel Hopper: If started with --hop, loops continuously through channels 1-13 via subprocess calls to iw.
  • Auto-reporter: Runs every 60 seconds (AUTO_SAVE_INTERVAL) to dump session tables into a daily summary CSV.

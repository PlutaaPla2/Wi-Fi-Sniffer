Passive Wi-Fi Sniffer Explained

This page explains how the Python script listens for Wi-Fi management frames, remembers each device by MAC address, estimates rough distance from signal strength, detects randomized MAC addresses, fingerprints packet Information Elements, prints a summary, and exports a CSV file.

The monitor-mode adapter receives frames already present in the air. The script reads metadata from those frames; it does not decrypt traffic.

Overview

test_sniffer_v0_7.py is a single-file Scapy program for passive Wi-Fi observation. It focuses on 802.11 management frames: Probe Requests from client devices and Beacons from access points are the most useful for this project.

What It Detects

It detects source MAC addresses, frame types, signal strength, probe SSIDs, access point SSIDs, randomized MAC status, vendor names for real MACs, and Information Element fingerprints.

What It Stores

It stores one record per MAC address in the global seen_devices dictionary. Each record accumulates packet count, first and last seen time, SSIDs, frame types, and signal-derived distance.

What It Produces

It prints live updates for interesting events, prints a final summary when the capture ends, optionally saves a PCAP, and always exports a timestamped CSV into ./csv_logs/.

How To Run

The script expects Linux, a compatible wireless adapter, monitor mode, and root permission. The setup comments at the top of the Python file show the intended commands. 

sudo ip link set wlan0 down
sudo iw dev wlan0 set type monitor
sudo ip link set wlan0 up
sudo python3 test_sniffer_v0_7.py --iface wlan0

Important Options

Option Meaning Example

Big Picture Flow

The program is easiest to understand as a pipeline. Packets enter from Scapy's sniff(), then handle_packet() decides whether the packet is useful and updates the global device table.

  1. Start command-line setup main() reads arguments, sets the distance environment, optionally starts channel hopping, and initializes timing.
  2. Start Scapy capture sniff() listens on the monitor-mode interface and calls a small nested handler() for each packet.
  3. Filter packet type handle_packet() ignores anything that is not an 802.11 management frame.
  4. Extract useful fields The script reads source MAC, RSSI, frame subtype, probe SSID, beacon SSID, and Information Elements.
  5. Update one device record The source MAC is the dictionary key. New MACs create a record; existing MACs update last seen time, packet count, RSSI, SSIDs, and fingerprint data.
  6. Finish and export When the capture stops, print_summary() prints the report and export_csv() writes a timestamped CSV.

Wi-Fi Basics

Wi-Fi frames have a type and subtype. This script only keeps management frames, because those are useful for discovering nearby clients and access points without joining any network.

Probe Request

A client device sends Probe Requests while scanning. Sometimes the SSID field is empty, meaning the device is asking for any nearby network. Sometimes it contains a known network name.

Example: a phone sends a Probe Request for Cafe_WiFi. The script adds Cafe_WiFi to that MAC's probe_ssids set.

Beacon

An access point sends Beacons repeatedly to announce that a Wi-Fi network exists. Beacon frames usually contain the AP's SSID.

Example: a router sends a Beacon with SSID Office_AP. The script classifies that MAC as an access point because beacon_ssid is present.

Frame Type Table Used By The Script

FRAME_TYPES maps management-frame subtype numbers to readable labels. Subtype 4 is Probe Request and subtype 8 is Beacon.

Data Model

The central structure is seen_devices. It is a dictionary keyed by source MAC address. This avoids double-counting the same MAC inside one capture session. 

seen_devices = {
  "aa:bb:cc:dd:ee:ff": {
    "first_seen": "14:32:10",
    "last_seen": "14:33:05",
    "rssi": -61,
    "distance_m": 1.2,
    "zone": "immediate  (<2 m)",
    "randomized": False,
    "vendor": "Apple, Inc.",
    "ie_fingerprint": "8a4d91c2",
    "frame_types": {"Probe Request"},
    "probe_ssids": {"Office_WiFi"},
    "packet_count": 7
  }
}

Why Sets Are Used

Fields such as frame_types, probe_ssids, and vendor_ies are sets. A set automatically removes duplicates, so the same SSID or frame type does not appear many times in the final report.

If a phone probes for HomeWiFi ten times, the set still stores it once: {"HomeWiFi"}. The packet count still increases, so you know how active the device was.

Distance Estimation

estimate_distance() converts RSSI into a rough distance. RSSI is signal strength in dBm. Values are usually negative: -40 dBm is strong, -80 dBm is weak.

distance = 10 ^ ((TX_POWER_DBM - rssi) / (10 * N))

Name Meaning In This Script
TX_POWER_DBM Expected RSSI at 1 meter. -59
rssi Measured signal from the current packet. Example: -65
N Path loss exponent. Higher means signal drops faster. 2.0 open, 2.7 office, 3.5 indoor

Worked Example

Suppose the script sees -65 dBm in the default office environment. Office uses N = 2.7. 

exponent = (-59 - -65) / (10 * 2.7)
         = 6 / 27
         = 0.222

distance = 10 ** 0.222
         = 1.7 metres approximately

The script then assigns a zone. Up to 2 meters is immediate, up to 7 meters is near, up to 20 meters is mid, and above that is far.

RSSI distance is only an estimate. Walls, adapter orientation, antenna quality, human bodies, and multipath reflection can move the estimate by several meters. Use it for proximity, not exact location.

MAC Analysis

A MAC address has 6 bytes. The script checks the second-lowest bit of the first byte. That bit is called the Locally Administered bit, often shortened to LA bit.

Byte 102
Byte 211
Byte 322
Byte 4-633:44:55
LA Bitset

Randomized MAC

is_locally_administered() returns True when the LA bit is set. That usually means the operating system generated a temporary randomized MAC address. 

mac = "02:11:22:33:44:55"
first_byte = int("02", 16)     # decimal 2
first_byte & 0b00000010        # non-zero, so randomized

Real Manufacturer MAC

If the LA bit is not set, the first three bytes are an OUI. The script checks KNOWN_OUIS first, then Scapy's conf.manufdb. This is done by lookup_vendor().

Example: 00:17:f2:12:34:56 starts with 00:17:f2. The curated table maps that OUI to Apple.
Vendor lookup is intentionally skipped for randomized MACs. A randomized MAC's first three bytes are not reliable manufacturer data.

IE Fingerprint

Information Elements, or IEs, are small tagged fields inside Probe Requests and Beacons. Their IDs and order are often influenced by the device's chipset, OS, and driver. The script uses that sequence as a fingerprint.

ID0
Length4
DataSSID
ID1
Length + DataRates

Why The Script Reads Raw Bytes

Scapy normally exposes IEs as chained Dot11Elt layers. But unusual, newer, vendor-specific, or truncated IEs can cause Scapy to stop building clean layers. To avoid losing data, the function get_ie_fingerprint() reads raw bytes from the first Dot11Elt and manually walks the TLV format. 

while i + 1 < len(raw_bytes):
    ie_id = raw_bytes[i]
    ie_len = raw_bytes[i + 1]
    data_start = i + 2
    data_end = data_start + ie_len
    ie_ids.append(ie_id)
    i = data_end

Readable Names And Hash

IE_NAMES maps common IE IDs to readable names. The raw ID list is also converted to an MD5 hash and shortened to 8 characters. The short hash is easier to scan in terminal output and CSV files.

Example IE IDs: [0, 1, 50, 45, 127, 221]
Readable sequence: SSID-Rates-ExtRates-HT-Cap-ExtCap-Vendor
Fingerprint: a short value such as 3f91a8c0.

For vendor-specific IEs, ID 221, the script also extracts the first three bytes of the IE data. Those bytes are vendor OUIs for the specific extension inside the frame.

Packet Handler

handle_packet(pkt) is the core of the program. Every captured packet passes through it.

  1. Keep only 802.11 management frames The handler returns early unless the packet has a Dot11 layer and pkt[Dot11].type == 0.
  2. Read source MAC pkt[Dot11].addr2 is treated as the device that sent the frame. Empty source addresses are ignored.
  3. Read RSSI If a RadioTap layer exists, the script reads dBm_AntSignal. Some adapters may not provide it.
  4. Extract SSID details Subtype 4 tries to read a Probe Request SSID. Subtype 8 walks Beacon IEs to find the AP SSID.
  5. Create or update record New MACs get a full record. Existing MACs update last_seen, RSSI, packet count, sets, and fingerprint fields.
  6. Print useful live output print_device_update() prints when a device is first seen or when a Probe/Beacon SSID adds new useful information.

Why This Avoids Some Double Counting

A device that sends 100 packets with the same MAC is still one entry, because the MAC is the dictionary key. The packet count increases, but the unique-device count does not.

If the same physical phone changes randomized MAC addresses during a scan, this script may count it as multiple MACs. The IE fingerprint can help identify that possibility, but the current unique-device count is still MAC-based.

Output

There are three output paths: live terminal updates, final terminal summary, and CSV export.

Live Update

First sighting prints MAC, frame type, signal, distance, MAC type, vendor when available, IE fingerprint, and first seen time.

Summary

print_summary() separates access points from client devices. APs have beacon_ssid; clients do not.

CSV Columns

CSV_FIELDS defines the exact CSV order. The CSV repeats scan start, stop, and duration on every row so each row stays useful even when filtered or copied elsewhere.

Column Group Fields Purpose
Scan timing scan_start, scan_stop, scan_duration Shows when the capture happened.
Identity mac, type, ssid, probe_ssids Describes the device or access point.
Signal rssi_dbm, distance_m, zone Supports proximity and occupancy decisions.
Analysis mac_randomized, vendor, ie_fingerprint, ie_sequence, vendor_ies Helps explain why a device was counted and whether the MAC is trustworthy.
Activity packet_count, frame_types, first_seen, last_seen Shows how active the device was during the scan.

Channel Hopping And Recovery

Channel Hopping

Wi-Fi devices transmit on different channels. If your adapter stays on one channel, it only sees traffic on that channel. With --hop, start_channel_hopper() runs in a daemon thread and repeatedly calls iwconfig <iface> channel <ch> for channels 1 through 13.

Hopping can see more devices, but it can also miss packets because the adapter is not listening to every channel at every moment.

Adapter Drop Recovery

Some USB adapters drop out of monitor mode. Scapy may return from sniff() without raising a normal exception. The script handles this by wrapping sniff() in a retry loop. 

ip link set <iface> down
iw dev <iface> set type monitor
ip link set <iface> up

MAX_RETRIES is 10 and RETRY_DELAY is 3 seconds. Already discovered devices stay in seen_devices across retries.

MAC A:
  02:11:22:33:44:55
  IE fingerprint: 8d31a2bf

MAC B:
  06:aa:bb:cc:dd:ee
  IE fingerprint: 8d31a2bf

Interpretation:
  They may be the same physical device using different randomized MACs,
  or they may be two similar devices with the same chipset/OS pattern.
  The script records the clue, but it does not merge them automatically.

Limits And Good Next Steps

Limits

Unique counting is MAC-based, so randomized MAC rotation can inflate counts. RSSI distance is approximate. Channel hopping can miss packets. Vendor lookup only works for real OUIs.

Good Next Steps

For room occupancy, compare packet count, recent activity, distance zone, and repeated IE fingerprints. Keep logs that explain why a MAC was counted or ignored so false counts can be debugged.

How To Read The Python File After This Page

Read the file in this order: constants and helper functions first, then handle_packet(), then print_summary() and export_csv(), and finally main(). That order follows the data flow from one packet to the final CSV.
